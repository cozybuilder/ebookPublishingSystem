# 공통 기준정보 (최신)

## 최종 방향 — Hybrid Publishing / Shorts Engine

이 시스템은 단순 조립 엔진이 아니라,

- **기본 = Content Assembly Engine** (운영비 0원)
- **선택 = API Auto Mode** (고급 기능)

으로 가는 **Hybrid** 엔진이다.

- 기본 사용자 : 파일 업로드 → 조립 → 완성
- 고급 사용자 : API Key 입력 → 자동 생성 → 조립 → 완성

## AI 생성과 프로그램의 역할

- 프로그램의 핵심은 생성(Generation)이 아니라 **조립(Assembly)과 완성(Finishing)** 이다.
- 기본 모드에서 자료(원고/대본/이미지/음성)는 사람이 준비한다.
- AI 생성은 **선택형 API Auto Mode** 에서만, 사용자가 자기 API Key 로 수행한다.

## 비용 원칙

- 자동화 = API 비용 발생
- Claude Pro/Max ≠ Claude API · ChatGPT Plus ≠ OpenAI API (웹 구독과 API를 혼동하지 않는다)
- 실제 추가 비용이 없을 때만 "공짜"라고 표현한다.
- **API 비용은 사용자 부담.**

## 두 모드 분리 원칙 (중요)

1. **Upload Mode** (기본) : 사용자가 파일 업로드 → 프로그램이 조립·편집·출력만 수행. 운영비 0원.
2. **API Auto Mode** (선택) : API Key 입력 시에만 자동 생성 사용.

규칙:

- **API Key가 없으면 절대 API를 호출하지 않는다.**
- **기본 플로우는 API 의존성이 전혀 없다.**
- 두 모드는 **UI / 코드 / 테스트에서 명확히 구분**한다.
- **API Key가 없어도 모든 테스트/빌드가 통과해야 한다.**

# ShortsFactory

**기본 모드 (Upload)**

- 대본 파일 업로드 / 이미지 업로드 / 음성 파일 업로드
- 프로그램: 컷 구성, 자막, BGM, 효과음, 화면 전환, ffmpeg 합성, 렌더링

**API Auto Mode (선택)**

- 주제 입력 → API로 대본/이미지/음성 생성 → 이후 자동 조립 및 렌더링

단, API Auto Mode는 **선택 기능**이다. 기본 제품은 파일 업로드 기반으로 동작해야 한다.

**운영비** : 0원 (ffmpeg 기반, 기본 모드)

# Ebook Publishing System

**기본 모드 (Upload)**

- Markdown 원고 업로드 / 이미지 업로드
- 프로그램: 챕터 구성, 표/체크리스트/인포그래픽 배치, 이미지 배치, 레이아웃·테마, PDF/DOCX/EPUB 생성, 미리보기, 크몽 샘플

**API Auto Mode (선택)**

- 주제 입력 → API로 원고 생성 (이미지 생성 가능) → 생성 자료를 엔진에 넘겨 PDF/DOCX/EPUB 출력

단, API Auto Mode는 **선택 기능**이다. 기본 제품은 Markdown + 이미지 업로드 기반으로 동작해야 한다.

**운영비** : 0원 (기본 모드)

# 개발 판단

- **src/studio 전부 삭제 금지.**
- 기존 생성기 코드는 **즉시 메인 플로우에 연결하지 말고**, 선택형 **API Auto Mode 후보 코드로 보존**한다.
- API 기능은 기본 플로우와 분리한다.
- API Key 없을 때는 모든 테스트/빌드가 통과해야 한다.

# 개발 방향

프로그램은 AI 서비스를 대체하지 않는다. AI가 만든(또는 사용자가 준비한) 결과물을 **상품 수준의 결과물로 완성하는 엔진**을 만든다. 기본은 조립 엔진, 선택적으로 API Auto Mode.

# 운영 원칙 — 개발자 자율성 / 승인 요청 최소화

원칙: **"중요한 결정은 사용자에게 묻고, 사소한 결정은 개발자가 책임지고 진행한다."**

- 다음은 개발자가 자율 판단으로 진행한다(승인 요청 없이) : 파일 생성·수정·삭제, 리팩토링, 테스트 실행, 빌드, 임시 파일 정리, 문서 업데이트, 경미한 버그 수정, 구조 개선.
- 다음은 **반드시 사용자에게 확인**한다 :
  1. 기능 추가 또는 제거
  2. 기존 동작 방식 변경
  3. 프로젝트 기준정보 변경
  4. 데이터 손실 가능성이 있는 작업
  5. 외부 서비스/API 비용 발생
  6. 보안 관련 작업
  7. 여러 선택지 중 방향 결정이 필요한 경우
- 작업 흐름을 끊는 반복 승인 요청을 지양하고, 가능하면 **구현 → 테스트 → 검증 → 완료 보고**로 한 번에 진행한다.
- 사용자는 CEO이며, 개발자는 멈춰서 반복 승인받기보다 **결과물과 변경사항 보고**를 우선한다.
